1. A Imagem Base

Evitando caixas-pretas de imagens pré-configuradas: o servidor web (Nginx) foi construído a partir de um S.O. limpo (ubuntu:latest).

Setup: Least Privilege

Executar containers constantemente com sudo é um risco. Adicionar o usuário ao grupo docker confere acesso ao socket do daemon sem expor o sistema inteiro ao nível root.

Dockerfile (Hardened)
FROM ubuntu:latest
# Evita travar automações de CI/CD aguardando input humano
ENV DEBIAN_FRONTEND=noninteractive
RUN apt-get update && apt-get install -y nginx
EXPOSE 80
# Força o Nginx no foreground (Evita que o container morra)
CMD ["nginx", "-g", "daemon off;"]

2. Isolamento & Limites

Subindo a aplicação isolada da rede principal e protegida estruturalmente contra ataques de exaustão de recursos (DoS).

Deployment Controls
# 1. Rede Customizada (Segurança por Isolamento)
$ docker network create practica1-net
# 2. Deploy com Limitações Estritas (Resource Limits)
$ docker run -d --name webapp-nginx \
  --network practica1-net \
  --cpus="0.5" -m="1g" \
  -p 8080:80 practica-1-nginx

Impacto das Restrições

Limite de 50% de CPU e 1GB de RAM. Se a aplicação sofrer um vazamento de memória ou ataque DDoS, ela não derrubará o servidor físico (Host).

3. A Muralha (mTLS)

Criação de um Proxy Reverso que só aceita conexões de clientes que possuam um certificado criptográfico válido (Mutual TLS).

1. PKI via OpenSSL Geração de Autoridade Certificadora (Root CA) e assinatura de certificados em pacote .p12 para o navegador do cliente.
nginx.conf (Proxy Settings)
# Hardening de Protocolos (Desabilita SSLv3, TLS 1.0/1.1)
ssl_protocols TLSv1.2 TLSv1.3;
# O coração do mTLS: Exigência de Validação
ssl_client_certificate /etc/nginx/ssl/ca.crt;
ssl_verify_client on;
# Redirecionamento Interno Seguro
proxy_pass http://webapp-nginx:80;

4. Orquestração Segura

Unificação do deploy via Docker Compose, garantindo a ordem lógica dos serviços e bloqueando escalação de privilégios.

docker-compose.yml
services:
proxy:
# O proxy acorda apenas quando o webapp estiver pronto
depends_on:
- webapp
# Hardening de Privilégios (A Pegadinha de Prova)
security_opt:
- no-new-privileges:true

No New Privileges

Impede que um atacante que invadiu o container como usuário comum utilize binários SUID (como sudo ou su) para virar root lá dentro. A permissão base é congelada permanentemente.

5. Auditoria de Imagens

Antes de aprovar para produção, a imagem Ubuntu-based foi escaneada em busca de CVEs (Vulnerabilidades e Exposições Comuns) utilizando o Grype.

Vulnerability Scanning
# 1. Execução do Scan exportando em JSON
$ grype practica-1-nginx -o json > grype.json
Parseamento Customizado (Python)

Em vez de ler o JSON cru, um script iterou os dados com lógica de negócio:

  • Filtrou apenas CVEs Medium, High e Critical.
  • Identificou o status "Won't Fix" (quando a falha existe, mas a fabricante não lançará patch).
  • Ordenou a matriz de forma decrescente pelo peso do risco.

A Jornada Continua

O que levo desta prática.

Build Próprio > Template Oficial

Subir um serviço partindo de um S.O. limpo garante previsibilidade. Você sabe exatamente o que está rodando, eliminando o inchaço e o risco de imagens "black-box".

Defesa em Profundidade

Segurança não é apenas o mTLS na porta da frente. É limitar a RAM (para evitar quedas de host), rodar em rede isolada e congelar privilégios com no-new-privileges. A barreira está em todas as camadas.
