Parte 2 - Hardening & IDS

A resposta ao incidente: Implementação de Suricata IPS, Patch Management e Auditoria Forense no ambiente previamente comprometido.

Blue Team Network Forensics System Hardening

1. Diagnóstico Inicial

A auditoria externa (feita do container Audit 10.10.10.50) revelou um ambiente com softwares legados e vazamento de informações.

Vulnerabilidades Críticas

  • ⚠️ Legacy Software: Apache 2.4.52 no host 10.10.10.10 (Vulnerável).
  • ⚠️ Server Tokens: Servidor expondo versão e S.O. nos headers HTTP.
audit@10.10.10.50
➜# nmap -Pn -sV --script vuln -p 80 10.10.10.10
➜# > CVE-2022-31813: Detected
➜# nmap -sC -sV -p 80 ...
➜# > http-server-header: Apache/2.4.52 (Ubuntu)

2. Defesa de Rede (IDS/IPS)

Implementação do Suricata no Web Server para detecção e bloqueio ativo de scans e ataques de negação de serviço (DoS).

Modo DROP Silencioso. Gera timeout no atacante.
Modo REJECT Responde com RST/ICMP. Corta conexão.
/etc/suricata/rules/local.rules
➜# # Detect Nmap Scan (Alert only)
➜# alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS 80 (msg:"Nmap Scan"; content:"Nmap"; ...)
➜# # Block ApacheBench DoS (Drop packet)
➜# drop tcp any any -> $HOME_NET 80 (msg:"DoS ApacheBench"; ...)
Suricata Traffic Analysis

3. Web Server Hardening

Mitigação das vulnerabilidades através de Patch Management (atualização via PPA externo) e ofuscação de headers no Apache.

Ação Corretiva

Repositórios padrão LTS frequentemente travam versões antigas. A solução exigiu adicionar repositórios PPA (Ondrej) para obter a versão de produção mais segura.

Patch & Config
➜# add-apt-repository ppa:ondrej/apache2 -y
➜# apt-get install apache2 -y (Update to > 2.4.60)
➜# vim /etc/apache2/conf-available/security.conf
➜# ServerTokens Prod (Oculta versão)
➜# ServerSignature Off (Remove rodapé)

4. SSH & Obscuridade

A auditoria anterior revelou o uso de "Porta Alta" (40022) como medida de segurança. O hardening real foi aplicado nas configurações de acesso.

❌ Porta 40022: Security by Obscurity (Ineficaz contra scans).
✅ sshd_config: Hardening real aplicado.
/etc/ssh/sshd_config
➜# PermitRootLogin no (Bloqueio de root remoto)
➜# MaxAuthTries 3 (Mitigação de Brute Force)
➜# AllowGroups sshusers (Princípio do Menor Privilégio)
➜# service ssh restart
Access Control Secure Config

5. Forense de Rede (PCAP)

Coleta e análise de tráfego para evidenciar conexões suspeitas vindas do container de auditoria (10.10.10.50).

// Wireshark Filter Logic
ip.src == 10.10.10.50 && tcp.flags.syn == 1
# Isola tentativas de conexão (SYN packets) vindas do IP atacante
Evidence Collection
➜# # Captura na porta 'escondida'
➜# tcpdump -i eth0 port 40022 -w /tmp/captura.pcap
➜# # Exfiltração para estação de análise
➜# docker cp container_web:/tmp/captura.pcap .
Tcpdump Wireshark

6. Auditoria Local (Lynis)

Scan automatizado de conformidade detectou falha grave de privacidade nos diretórios de usuários (World Readable).

Diagnóstico (Antes)

drwxr-xr-x (755) /home/administracion
Qualquer usuário logado podia ler arquivos alheios.

Correção de Permissões
➜# ./lynis audit system --quick
➜# > WARNING: Permissions of home directories
➜# sudo chmod 700 /home/administracion
➜# ls -ld /home/administracion
➜# > drwx------ (700) # Acesso restrito ao dono
Least Privilege Compliance
