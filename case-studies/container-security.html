1. A Imagem Base

Evitando caixas-pretas de imagens pré-configuradas: o servidor web (Nginx) foi construído a partir de um S.O. limpo (ubuntu:latest).

Setup: Least Privilege

Executar containers constantemente com sudo é um risco de segurança. Adicionar o usuário ao grupo docker confere acesso ao socket do daemon sem expor o sistema inteiro ao nível root (/var/run/docker.sock).

Dockerfile (Hardened)
FROM ubuntu:latest
# Evita travar automações de CI/CD aguardando input humano
ENV DEBIAN_FRONTEND=noninteractive
RUN apt-get update && apt-get install -y nginx
EXPOSE 80
# Força o Nginx no foreground (Evita que o container morra)
CMD ["nginx", "-g", "daemon off;"]

2. Isolamento & Limites

Subindo a aplicação isolada da rede principal e protegida estruturalmente contra ataques de exaustão de recursos (DoS).

Deployment Controls
# 1. Rede Customizada (Segurança por Isolamento)
$docker network create practica1-net
# 2. Deploy com Limitações Estritas (Resource Limits)
$docker run -d --name webapp-nginx \
  --network practica1-net \
  --cpus="0.5" -m="1g" \
  -p 8080:80 practica-1-nginx

Impacto das Restrições (Cgroups)

Limite de 50% de CPU e 1GB de RAM. Se a aplicação sofrer um vazamento de memória (Memory Leak) ou ataque DDoS, ela será estrangulada pelas restrições e não derrubará o host docker subjacente.

3. A Muralha (mTLS)

Criação de um Proxy Reverso que só aceita conexões de clientes que possuam um certificado criptográfico válido (Mutual TLS).

PKI via OpenSSL

Geração de Autoridade Certificadora Privada (Root CA) e assinatura de certificados exportados em formato .p12 para validação bidirecional direto no navegador do cliente.

nginx.conf (Proxy Settings)
# Hardening de Protocolos (Desabilita SSLv3, TLS 1.0/1.1)
ssl_protocols TLSv1.2 TLSv1.3;
# O coração do mTLS: Exigência rigorosa de Validação
ssl_client_certificate /etc/nginx/ssl/ca.crt;
ssl_verify_client on;
# Redirecionamento Interno Seguro
proxy_pass http://webapp-nginx:80;

4. Orquestração Segura

Unificação do deploy via Docker Compose, garantindo a ordem lógica dos serviços e bloqueando escalação de privilégios.

docker-compose.yml
services:
proxy:
# O proxy acorda apenas quando o webapp estiver pronto
depends_on:
- webapp
# Hardening de Privilégios
security_opt:
- no-new-privileges:true

No New Privileges

Essa flag é vital. Ela impede que um atacante que invadiu o container como usuário comum utilize binários SUID (como sudo ou su) para virar root lá dentro. A permissão base é congelada permanentemente no momento da inicialização.

5. Auditoria de Imagens

Antes de aprovar para produção, a imagem Ubuntu-based construída foi escaneada em busca de CVEs utilizando o Grype.

Vulnerability Scanning
# Execução do Scan exportando resultados em JSON para pipeline
$grype practica-1-nginx -o json > grype.json

Parseamento Customizado (Python)

Em vez de ler o JSON cru, um script iterou os dados com lógica de negócio específica para este ambiente:

  • Filtrou alertando apenas sobre CVEs de severidade Medium, High e Critical.
  • Identificou o status "Won't Fix" (quando a falha existe na distro, mas a fabricante não lançará patch, exigindo mitigação manual).
  • Ordenou a matriz de forma decrescente pelo peso do risco para priorização das equipes de AppSec.

Defesa em Profundidade

Segurança em containers não é apenas colocar um mTLS na porta da frente. É limitar recursos vitais (CPU/RAM), isolar o roteamento interno e congelar privilégios com diretivas do kernel (cgroups e namespaces). A barreira está em todas as camadas.
