Estudo de Caso: Hospital Esperanza Libre
Parte 2 - Hardening & IDS

A resposta ao incidente: Implementação de Suricata IPS, Patch Management e Auditoria Forense no ambiente previamente comprometido.

Network Forensics System Hardening Suricata IPS

1. Diagnóstico Inicial

A auditoria externa (feita do container Audit 10.10.10.50) revelou um ambiente com softwares legados e vazamento de informações essenciais.

Vulnerabilidades Críticas Identificadas

  • Legacy Software: Apache 2.4.52 no host 10.10.10.10 (Vulnerável a CVEs conhecidas).
  • Server Tokens: Servidor expondo versão e sistema operacional diretamente nos headers HTTP.
audit@10.10.10.50
➜#nmap -Pn -sV --script vuln -p 80 10.10.10.10
> CVE-2022-31813: Detected
➜#nmap -sC -sV -p 80 ...
> http-server-header: Apache/2.4.52 (Ubuntu)

2. Defesa de Rede (IDS/IPS)

Implementação do Suricata no Web Server para detecção e bloqueio ativo de scans e ataques de negação de serviço (DoS).

Modo DROP Abordagem furtiva e silenciosa. O pacote é descartado sem notificar o remetente, forçando um timeout no lado do atacante.
Modo REJECT Responde ativamente com RST (TCP) ou ICMP Port Unreachable. Corta a conexão instantaneamente.
/etc/suricata/rules/local.rules
# Detect Nmap Scan (Alert only)
➜alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS 80 (msg:"Nmap Scan"; content:"Nmap"; ...)
# Block ApacheBench DoS (Drop packet)
➜drop tcp any any -> $HOME_NET 80 (msg:"DoS ApacheBench"; ...)

3. Web Server Hardening

Mitigação das vulnerabilidades através de Patch Management (atualização via PPA externo) e ofuscação de headers no Apache.

Ação Corretiva

Repositórios padrão LTS frequentemente travam versões antigas por estabilidade. A solução exigiu adicionar repositórios PPA (Ondrej) para obter a versão de produção mais segura e recente.

Patch & Config
➜add-apt-repository ppa:ondrej/apache2 -y
➜apt-get install apache2 -y // Update to > 2.4.60
➜vim /etc/apache2/conf-available/security.conf
ServerTokens Prod // Oculta versão
ServerSignature Off // Remove rodapé

4. SSH & Controle de Acesso

A auditoria anterior revelou o uso de "Porta Alta" (40022) como medida de segurança. O hardening real precisa ser aplicado nas diretivas de acesso do serviço.

Porta 40022 Security by Obscurity (Ineficaz contra nmap scans)
sshd_config Políticas reais de restrição aplicadas
/etc/ssh/sshd_config
PermitRootLogin no // Bloqueio de root remoto
MaxAuthTries 3 // Mitigação de Brute Force
AllowGroups sshusers // Princípio do Menor Privilégio
➜service ssh restart

5. Forense de Rede (PCAP)

Coleta e análise de tráfego (packet capture) para evidenciar conexões maliciosas vindas do container de auditoria na rede do hospital.

// Wireshark Filter Logic
ip.src == 10.10.10.50 && tcp.flags.syn == 1
Isola as tentativas agressivas de conexão (pacotes SYN isolados) partindo diretamente do IP da máquina atacante.
Evidence Collection (Tcpdump)
# Captura na porta de gerência escondida
➜tcpdump -i eth0 port 40022 -w /tmp/captura.pcap
# Exfiltração segura da evidência para o host de análise
➜docker cp container_web:/tmp/captura.pcap .

6. Auditoria Local (Lynis)

Scan automatizado de conformidade detectou falha grave de privacidade nos diretórios de usuários, permitindo que chaves SSH fossem lidas (como ocorreu no ataque).

Diagnóstico de Falha (Antes)

drwxr-xr-x (755) /home/administracion

Qualquer usuário logado, incluindo o serviço web comprometido (www-data), podia ler arquivos alheios.

Correção de Permissões (Compliance)
➜./lynis audit system --quick
> WARNING: Permissions of home directories
➜sudo chmod 700 /home/administracion
➜ls -ld /home/administracion
> drwx------ (700) // Acesso restrito exclusivo ao dono

Resultados Obtidos

Defesa Ativa

Configuração bem-sucedida de IPS (Suricata) bloqueando ataques de DoS e Fuzzing na camada de rede, antes mesmo de atingirem a aplicação web.

Hardening & Compliance

Servidor atualizado via repositório seguro, SSH blindado contra força bruta e permissões de arquivos corrigidas rigorosamente conforme as políticas do Lynis.

final_report_status
